## 🎯 模块一：项目实干经历（用这套话术做自我介绍）

"我在该项目中**从底层单片机外设驱动到上层核心飞控闭环控制算法，完全独立开发了一套硬实时的高性能无人机飞行控制器（C语言裸机架构）**。
该飞控没有依赖厚重的 RTOS 系统，而是基于 STM32G431 的高优先级外部中断和 DMA，利用 SPI 总线与 ICM42688 陀螺仪构建了严苛的 **2000Hz (0.5ms)** 串级 PID 控制闭环。并自主实现了基于 FFT 的动态陷波滤波器对传感器进行 DSP 数字去噪，解决了大尺寸电机狂暴共振的难题。并在执行层完成了 DShot 数字协议与经典 PWM 的双重电机驱动重构。"

### 核心技术点梳理表
| 岗位对应需求 | 项目中使用的技术方案及原理 |
| :--- | :--- |
| **单片机/ARM 开发** | 基于 Cortex-M4 (STM32G431) 和 HAL 库。外设调用涵盖 SPI (DMA), I2C, 定时器(TIM) 输入捕获及多路 PWM 输出, 独立看门狗 (IWDG), 并由 NVIC (中断向量表) 进行抢占式最高优先级控制。 |
| **飞控算法** | 采用 Madgwick 互补滤波四元数姿态解算，规避万向节死锁。控制回路运用“角度-角速度”非线性串级 PID。并引入了 Anti-Windup (抗积分饱和) / TPA 油门衰减保护 / 反重力增强等高级现代飞控算法特性。 |
| **数字通信** | 接收端实现 PPM / SBUS 高频率硬解；输出端不仅支持模拟 PWM 制式，也完整实现了基于 DMA 的 DShot600 无损高频数字通讯协议，避免了模拟信号在长走线中的电磁干扰，大幅提高了电机响应分辨率。 |
| **软件架构与抗干扰** | 采用硬实时架构（Hard Real-Time），依托芯片引脚级的中断触发机制(EXTI DRDY)；引入第一阶与第二阶的低通滤波器 (PT1/PT2)，并搭载高阶 DSP 计算出动态 FFT 陷波中心阻滞频段。 |

## 💡 模块二：高频硬核预测题（降维打击版回答）

### 题目一：既然你是独立写出的飞控，请谈谈你在写飞控时，是如何保证 2000Hz 控制回路的绝对实时性的？为什么不用 FreeRTOS？

**【考察点】：** 裸机开发的高级架构能力、中断/DMA 的底层理解。

**【面试官满分回答】：**
"因为飞控的串级 PID 中的积分和微分计算，对环路时长（`dt` 的绝对刚性）有极其严苛的要求。
我在架构时**完全抛弃了主循环阻塞（while+delay）和 RTOS 的软时间片切换**，因为 RTOS 哪怕优化得再好，任务上下文切换（Context Switch）的开销也有十几微秒，且容易引发优先级翻转导致延时抖动（Jitter）。
我采用的是纯硬件触发机制：我将 ICM42688 陀螺仪配置为输出 2000Hz 的 DRDY (Data Ready) 同步引脚脉冲。一旦触发 STM32 的 EXTI 最高级外部中断，CPU 会立刻响应，并**连同 DMA 一起光速搬运 SPI 数据，原地在中断服务函数里解算四元数与执行 3 个轴系的内外环 PID，随后通过 Mixer 混控直达定时器输出给电机**。这套‘一推到底’的超短数据流，保证了核心算力的微秒级绝对同步。对于像读取气压计、处理串口发点遥控指令这些‘慢速杂活’，我则把它们放在被底层中断抢占的最底层 while 循环里慢慢做。这样兼顾了控制回路的绝对暴力和软件拓展的柔性。"

### 题目二：你在调参大机架四轴时，一定遇到过电机的巨大震动。你是怎么处理震波导致 PID 中“D 狂躁”甚至烧电机问题的？

**【考察点】：** 真实无人机世界的陷阱（噪声）、DSP 信号处理学知识（滤波器）。

**【面试官满分回答】：**
"是的，大机架（比如装配大桨的 F450 以上）物理机械谐振通常落在 80~150Hz。这些由陀螺仪采样进来的高频高副震动噪音，一旦经过 PID 的 D 项（微分放大），会变成极其夸张的狂躁反相指令，让电机高抛甚至拉烟烧毁。
我的做法不是去把 D 调到 0 妥协，而是在软件算法端筑起了两道高墙：
第一道是低通防线，对于原始生肉数据我引入了 **PT1 或 PT2 的软件低通滤波 (Low-Pass Filter)**，把高于 90Hz 这类显而易见的非人为物理翻滚动作给大刀劈掉。
第二道是精准打击，我借助芯片内部的 FPU 和 Cortex DSP 库，在后台微弱的时间片里跑 **FFT（快速傅里叶变换）** 算法，动态侦测噪音能量最集中的频段在哪里。然后用这个中心频率算出一组 **二阶的双二重数字陷波滤波器 (Notch Filter) 系数**，像手术刀一样对准那个频率切下去！这样它既杀灭了共振频率里的毒刺，又绝对不影响相邻真实控制信号本身的相伴延迟感（大幅优于纯低通）。这套技术是现代 Betaflight 等顶级赛用固件的核心壁垒。"

### 题目三：你的简历里写了抗积分饱和（Anti-Windup）。能举个无人机真正飞行中必须用到这个机制的场景吗？你是怎么写的？

**【考察点】：** PID 深层理论与代码落地的绑定能力。

**【面试官满分回答】：**
"比如一架正在树丛上方悬停的飞机，因为一阵极强的下洗气流（风）被一直向下压。此时，系统存在高度差，但由于此时飞控判断哪怕四台电机全部满负荷运作，也顶不过这阵风。
如果没有 Anti-Windup 机制，系统内的 I 项（纯数字积分项）会在漫长的数秒内不断疯狂累加这些达不到的高度差，积攒出一个非常恐怖且不着边际的错误天文数字。等风刚刚停歇的瞬间，这巨大的残余 I 项就像一颗出膛的炮弹，瞬间把飞机的油门再次锁到 100% 直冲云霄（OverShoot），这就是非常致命的“积分过饱和”。
我的代码里，做了一段叫做**输出预估（Pre-Actuator Saturation Test）**。我会算出当前 PID 的总力，并和电调的机械阈值作对比。如果发现推子已经推到底（比如 PWM 已经被我占到了 2000 的极值），而偏差依然想让飞机朝这个出力的方向继续前行，此时我会**从数学上硬性冻结当前轴的 I 项数组，严禁它再进行 `+=(误差*dt)` 的动作**。等外力消散、推力重新回到受限域内部时，才重新恢复它的积分。从根源上斩断了“死亡抛升”。"

### 题目四：看你也实现了 DShot 数字电调协议，比起以前咱们在航模上用的 PWM，它具体强在哪里？单片机怎么实现这种协议的发包？

**【考察点】：** 现代通信协议深度、驱动协议开发能力。

**【面试官满分回答】：**
"PWM 是个上世纪八十年代的模拟产物，靠的是高频脉冲宽度的‘时长’（1~2ms）来确定油门位置。它的毛病在于模拟量极其容易受到电源干扰和磁场噪声的挤压（使得微小油门不均匀），而且必须麻烦地人工调校电调的最大最小行程。
而 **DShot（例如 DShot600）是一种数字编码协议，且刷新率高达数万 Hz。**它每一帧直接通过 `0` 和 `1` 发送 `(11位油门值 + 1位遥测包请求位 + 4位CRC校验码)`，在总共 16 Bit 这个绝对确定性的数字字典里，没有任何干扰的生存空间。
我在单片机里实现它时，其实靠的是一个精妙的“硬件欺骗”技术。因为芯片没有原生的 DShot 模块单元，我是依靠配置定时器（TIMD）的 `PWM Burst Mode`（批量调制模式），配合 DMA 的存储器到外设的高强输送。把 `0` 定义为一个短占空比的脉冲（例如 37.5%），把 `1` 定义为一个长占空比的脉冲（75%）。随后将要发送的字组打散成内存里的一个个 PWM 数组。最后只需呼叫一声 DMA 自动倾倒并重启波形发声器，DShot 信号就会被完美的射往四通电调上。"
